# Threat Model

A focused STRIDE walk-through of an SDN OpenFlow control plane, written specifically for this project's deployment shape — a single POX controller speaking OpenFlow 1.0 to Open vSwitch instances, with the entropy + PCA + RandomForest detector chain landing `ofp_flow_mod` drop rules in response to volumetric floods. The goal is to be explicit about what this project addresses, what it doesn't, and where the SDN-specific risks differ from a traditional layer-2 / layer-3 enterprise switch deployment.

References: NIST SP 800-94 (*Guide to Intrusion Detection and Prevention Systems*), MITRE D3FEND for defensive-technique mapping.

## The SDN asymmetry

A traditional layer-2 broadcast storm or layer-3 volumetric flood targets a single switch or a single victim host. Storm-control and inbound ACLs (the GTA / UMD IT-20 toolkit this project's README maps to its SDN equivalent) are local mitigations: the affected switchport drops or rate-limits, the rest of the network keeps forwarding. An SDN flood targets a fundamentally different surface — **the controller's `PACKET_IN` queue depth**. Every new `[srcip, dstip]` pair the switch hasn't seen before generates a `PACKET_IN` event up the southbound channel, and the controller has a bounded rate at which it can install matching flow-table rules. When the attacker bursts enough new pairs to saturate that rate, *every* flow in the network — not just the attacker's — falls behind whatever the controller can process. The single point of failure is no longer the wire, it's the control plane.

The mitigation primitive this project ships — an `ofp_flow_mod(command=OFPFC_ADD, match=ofp_match(in_port=port, nw_src=...), actions=[], hard_timeout=30, priority=OFP_DEFAULT_PRIORITY+1)` drop rule installed at `pox_controller.py:54` — is **not** the same as a traditional inbound ACL. It is the controller *teaching every switch in the data plane* to stop sending it the offending PACKET_INs. After installation the switch silently drops matching packets at line rate without consulting the controller, which restores `PACKET_IN` queue depth for the rest of the network. The `priority` above default ensures the drop outranks the L3 learning-switch forwarding rule the controller installed for the same flow earlier; the per-key `nw_src` retention in `port_stats` means a returning attacker after `hard_timeout` expires re-trips the threshold one window later, not `port_count_threshold` packets later.

## STRIDE on the southbound channel

OpenFlow 1.0 runs over TCP port 6633 by default, unencrypted. Mininet's `--controller=remote,ip=...,port=6633` exposes this channel to any host that can reach it, and the cleartext default is the assumption this project's threat model treats most carefully.

| STRIDE | Threat | This project | MITRE D3FEND mapping |
|---|---|---|---|
| **S**poofing | An attacker speaks OpenFlow to the switches, masquerading as the controller. Can rewrite arbitrary flow-table rules — equivalent to root on every switch. | Out of scope. Standard mitigation is TLS on the southbound channel (POX supports it via `--cert`/`--key`; Mininet's default `--controller=remote` does not enforce TLS). The threat-model section of the README acknowledges this is the highest-impact unaddressed gap. | D3FEND `Mutual Authentication` (D3-MA), `Strong Password Policy` (D3-SPP) — neither implemented here. |
| **T**ampering | Flow-mod payloads modified in transit between controller and switch. | Out of scope. Mitigation: TLS again. The integrity property of the southbound channel reduces to the encrypted transport. | D3FEND `Message Authentication` (D3-MAN). |
| **R**epudiation | A switch could deny it received a specific flow-mod. | Out of scope. Mitigation is signed OpenFlow events, which is essentially nonexistent in practice — no major controller ships it, and POX's pipeline does not provide hooks for cryptographic signing of `ofp_flow_mod` payloads. The threat is real (an attacker who has briefly impersonated the controller could disclaim flow-table changes later) but the industry-standard answer is operational logging, not cryptographic non-repudiation. | D3FEND `Sender Reputation Analysis` (D3-SRA). |
| **I**nformation disclosure | By default, on a table-miss the switch forwards the **full packet** (including payload) up to the controller as a `PACKET_IN`. A malicious or compromised controller can therefore read application-layer data for any flow that misses the table. | Partially addressed. The detector chain in this project does not log payload bytes — it only computes per-window statistics on header fields (`dst_ip`, `src_ip`). But a different controller module loaded into the same POX runtime could. The hardening primitive is `OFPCML_MAX` or a configured short `miss_send_len` so only headers reach the controller; this project ships POX defaults. | D3FEND `Inbound Traffic Filtering` (D3-ITF) — partially through the entropy threshold. |
| **D**oS of the controller | The flood case this project *is* designed for. Volumetric flood toward any victim drives the controller's `PACKET_IN` rate past sustainable, after which the entire network's flow-installation latency rises. The detector chain catches it and the `ofp_flow_mod` drop rule offloads suppression to the switch. | **Addressed.** The headline deliverable. The detector chain (entropy + PCA + RF) is calibrated specifically for this threat; the `priority+1` and `hard_timeout=30` choices in `pox_controller.py:check_ddos` are tuned for fast recovery without permanent state changes that would complicate operator response. | D3FEND `Network Traffic Filtering` (D3-NTF), `Inbound Traffic Filtering` (D3-ITF). |
| **E**levation of privilege | A crafted OpenFlow message that exploits a parser bug in POX or OVS could grant the attacker code execution in either process. | Out of scope. The mitigation is keeping POX and OVS current; this project pins POX via git tag in the Phase 3 Dockerfile to make the runtime version reproducible. CVE monitoring is operational, not architectural. | D3FEND `Software Update` (D3-SU). |

NIST SP 800-94 §6 frames the distinction between detection (recognizing the attack is happening) and prevention (stopping it). This project's entropy + PCA + RF chain is a **detector**; the `ofp_flow_mod` drop rule is the **prevention** primitive the detector triggers. SP 800-94 §6 makes the case that IDS-only deployments incur false-positive cost the operator has to absorb, while IPS deployments incur attack-on-the-controller-itself risk if the prevention rule is mis-applied. This project's `hard_timeout=30` ceiling is the answer to that trade-off: a wrongly-installed drop rule self-expires within 30 seconds.

## Scope: what is *not* addressed

To keep the threat model honest about what this project ships, the following are explicitly out of scope. Each one is a real risk; this project simply doesn't claim to address it.

- **Southbound TLS.** The deployment ships POX in `--controller=remote` plaintext mode. Adding `--cert`/`--key` to the POX command line and a corresponding CA pin on Mininet/OVS is straightforward; it has not been implemented here because the synthetic test environment is single-host and the threat model focuses on the volumetric-DoS attack surface.
- **Controller authentication.** No authentication on the controller's northbound REST/CLI surface. POX itself exposes none by default; if this controller were extended with a REST-driven config knob (e.g. for a network operator to tune `port_count_threshold` live), that endpoint would need its own auth + audit story.
- **Low-and-slow / application-layer attacks.** Slow-loris, HTTP/2 RST flood, application-layer Layer-7 attacks. The detector chain is calibrated against window-level entropy + per-flow features and does not retain enough state to recognize multi-minute, low-rate patterns. Phase 4 §3.10 (multi-feature detector) is the natural extension here but is not in Phase 3.
- **Multi-controller coordination.** The companion report (Chapter 7) and v3 §4.11 propose distributed multi-controllers as the natural next step; not in Phase 3. A flood that's split across multiple controllers' partitions could evade single-controller detection.
- **Data-plane attacks below the entropy detector's window size.** Bursts smaller than 250 packets, even at high rate, may not close a window before they end. This is a deliberate design trade-off: smaller windows produce noisier entropy estimates and more false positives.
